## Detailed Findings

### 🔴 1. `useDebug.ts` — `encodeMemory` creates a throwaway store on every call

**File:** `src/hooks/useDebug.ts` (line ~162)

```ts
const memory = new InMemoryEpisodicStore().encode({...})
```

The `encodeMemory` callback creates a **brand new** `InMemoryEpisodicStore` instance for every perception event purely to call `.encode()`. A new `Map` is allocated and the constructor runs, then the instance is immediately discarded after GC.

**Impact:**
- Unnecessary GC pressure on every perception event (potentially 10-30Hz)
- `encode()` internally calls `nextMemoryId()` which increments a module-level counter, meaning ID generation isn't isolated to a single store — if this and the BT's real store interleave, IDs may not reflect a sequential order within each store.

**Fix:** Extract `encode()` to a standalone function, or keep a singleton store instance inside the hook.

---

### 🟡 2. `episodic-store.ts` — `recall()` has an ambiguous overloaded return type

**File:** `src/memory/episodic-store.ts` (line ~72)

```ts
recall(queryOrId: RecallQuery | string): RecallResult | void
```

The method does two different things:
- **String argument:** looks up by ID, mutates `recallCount`/`relevance` in-place, returns `undefined`
- **Query argument:** filters, sorts, slices, mutates results, returns `RecallResult`

**Problems:**
- Calling code has to check `if (!result)` and cannot distinguish "no match" from "recalled by ID"
- TypeScript cannot narrow the return based on the argument shape at the call site without user-defined type guards
- Mutating memory state under a "recall" name is surprising (caller might expect a read-only lookup)

**Suggestion:** Add a separate `recallById(id: string): EpisodicMemory | null` method instead of overloading.

### 🟡 3. Duplicated forgetting curve computation

**File:** `src/hooks/useDebug.ts` (line ~237)

```ts
function computeRelevanceDebug(
  ageMs: number, salience: number, recallCount: number
): number { ... }
```

This is an **identical copy** of `computeRelevance()` from `episodic-store.ts` (line ~20), which is not exported. Maintaining two copies guarantees they'll drift.

**Fix:** Export `computeRelevance` from `episodic-store.ts` and import it in `useDebug.ts`.

### 🟡 4. Debug panel maintains its own independent memory list

**File:** `src/hooks/useDebug.ts` — `encodeMemory`, `purgeMemories`, `simulateForgetting`

The BT engine maintains its store on `bb.episodicStore`, while the debug hook maintains a separate `memories[]` array in React state. When the BT encodes a memory via `memorize.episodic`, the debug panel will only see it if a matching `perception` tracer event is also emitted. This means:

- If the BT encodes a memory programmatically (not via perception), the debug panel won't show it
- The debug panel's memory count can diverge from the BT's actual memory count
- `purgeMemories` in debug only affects the local state copy, not the BT's store

This is arguably intentional (decoupling debug from engine state), but it's fragile and undocumented.
